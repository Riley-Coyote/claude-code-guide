Section 01

From Spark to Shipped

Opening Hook: Building with Claude is different. You're not just writing code — you're orchestrating an AI that generates, tests, fixes, and ships autonomously.
Every great project follows the same arc: you start with a spark of an idea, you explore what's possible, you lock down a design, you build like hell, you test ruthlessly, and then you polish until it shines. The difference with Claude Code is that Claude handles the execution — you focus on the thinking and the decisions.

This guide maps that entire journey. You'll learn how to structure a project so Claude can work autonomously from Phase 1 (idea to design) through Phase 2 (code to verified) to Phase 3 (polish to shipped). Every pattern here is battle-tested.

Section 02

Three Phases, One Journey

The Build Flow at a Glance
Phase 1: Design
→
Phase 2: Build
→
Phase 3: Polish
Phase 1 (Design) is where you think. You and Claude work together to nail down what you're building, the architecture, the constraints, and the exact acceptance criteria. Output: a CLAUDE.md file that defines everything.

Phase 2 (Build) is where Claude works. It receives task briefs, generates code, runs verification, fixes errors, and commits. You're in the loop but the AI drives. Output: a working implementation that passes all checks.

Phase 3 (Polish) is refinement. Edge cases, performance, documentation, and the final review before shipping. Output: a production-ready codebase.
Section 03

Everyone Builds Differently

Here's a Framework That Works
There's no single "right way" to code. But for autonomous Claude execution at scale, a few patterns matter: machine-readable verification (so Claude knows exactly what failed), self-contained task briefs (so Claude doesn't have to hunt for context), cross-session state files (so work persists across context resets), and hard constraints (so Claude doesn't drift).

This guide gives you those patterns. Apply them, adapt them, make them your own. The core is flexible — use it as a starting point, not a straitjacket.

You'll see how to structure it, how Claude builds it autonomously, and how the three phases play out in practice. The patterns scale from a single script to a full-featured application.

Phase 1

Claude Chat: Your Thinking Partner

The Phase 1 workflow: you and Claude collaborate to nail down the entire project before any code is written.
Phase 1 is a conversation. You bring the idea, Claude brings the execution expertise. Together you answer the hard questions: What exactly are we building? What's in scope and what's deferred? What's the architecture? What are the non-negotiables? This phase ends when you have a CLAUDE.md file that Claude can follow autonomously.
Phase 1

Phase 1: Spark

From idea to hypothesis
What Happens
Start with a one-sentence problem statement. TaskFlow: "Teams need a fast way to visualize their project workflow." That's it. Claude asks clarifying questions: Who's the user? What's the use case? What problem does it solve?
Your Deliverable
A Clear Problem Statement
One paragraph that tells Claude who the users are, what problem they have, and why they can't use existing tools.
Phase 1

Phase 1: Foundations

From hypothesis to architecture
What Happens
With the problem locked down, Claude asks about the technical foundation. What platform? What language? What database? What integrations? TaskFlow: web app → React frontend + Node.js backend. Postgres for persistence. Real-time via WebSockets. These decisions ripple through the entire build.
Your Deliverable
Tech Stack & Architecture Decisions
The platform, language, key libraries, database choice, and deployment target. Not "maybe use Redis" — concrete decisions that Claude can build toward.
Phase 1

Phase 1: Design Lock

From architecture to specification
What Happens
Now Claude asks for the design constraints. What screens? What data model? What API endpoints? What's in Phase 1 vs. Phase 2? TaskFlow Phase 1: login, project creation, drag-drop board, WebSocket sync, no AI recommendations (that's Phase 2). This locked-down scope is critical.
Your Deliverable
Scope & Design Spec
List of screens, API endpoints, data model sketch, and explicit scope boundaries. "What we're NOT building in Phase 1" is as important as what we are.
Phase 1

Phase 1: Rapid Prototype

From specification to visual mockup
What Happens
Claude creates a quick mockup of the UI. Not pixel-perfect — just enough to validate the flow with you. TaskFlow: login page, project list, board with drag-drop zones. You give feedback: change the color, move this button, add this button. The mockup evolves in real-time.
Your Deliverable
UI Mockups (HTML or Figma)
Quick visual mockups that show the user flow. Use Claude's sketch skill or Figma — doesn't matter. The point is visual alignment before any real development.
Phase 1

Phase 1: Launch Planning

From mockup to launch criteria
What Happens
What does "done" mean? TaskFlow Phase 1: all endpoints implemented, 85% test coverage, zero TypeScript errors, login works with test users, board renders and syncs in under 200ms. Claude asks for these constraints up front so it knows exactly what to verify against.
Your Deliverable
Acceptance Criteria & Launch Checklist
Explicit criteria: performance targets, test coverage %, API response times, browser compatibility, accessibility standards. Make it measurable.
Phase 1

Phase 1: Ready to Code

From launch criteria to CLAUDE.md
What Happens
You and Claude distill everything above into a CLAUDE.md file. This is the bootstrap file — Claude reads it automatically at session start and knows exactly what to build. It includes: project identity, tech stack, architecture, scope, acceptance criteria, code style rules, and the exact build order for Phase 2.
Your Deliverable
CLAUDE.md: The Execution Blueprint
A markdown file that captures everything: project description, hard constraints, architecture decisions, build steps, code style guide, acceptance criteria, and the autonomous execution protocol.

TASKFLOW EXAMPLE: See the full TaskFlow CLAUDE.md with tech stack (React, Node, Postgres), build steps (setup → API → frontend → integration → testing), and the exact verification protocol Claude will follow in Phase 2.

Phase 1-2

Phase 1 → Phase 2: The Handoff

From design to code generation
At the end of Phase 1, you give Claude one instruction: "Run /build-all". That slash command exists in .claude/commands/build-all.md. It tells Claude to read CLAUDE.md, read PROGRESS.md (which tracks what's been done), and start executing the build order step by step. Phase 2 is fully autonomous.

Key insight: Phase 1 is about making decisions. Phase 2 is about execution. The handoff is clean — you've decided everything, Claude executes everything. No back-and-forth needed in Phase 2 unless something breaks.

Phase 2

Phase 2: Code Generation

The BUILD loop: Claude receives a task brief and generates implementation
What Happens
Claude reads the first task brief from docs/tasks/task-01-*.md. The brief is self-contained: here's what to build, here are the file paths, here are the constraints, here's what "done" looks like. Claude generates the full implementation, then moves to verification.
Why Task Briefs Matter
A task brief is not a casual request. It's a formal specification that a subagent Claude instance reads once and implements completely. Include: exact scope, file paths, relevant type definitions inlined, constraints from CLAUDE.md, and acceptance criteria. The subagent shouldn't need to read anything else.
Phase 2

Phase 2: Autonomous Loop

The VERIFY → FIX → VERIFY cycle
BUILD
→
VERIFY
→
FIX
→
VERIFY
→
COMMIT
After implementing, Claude runs a verification script: bash scripts/verify/verify-step-01.sh. The script outputs structured JSON: which checks passed, which failed, and why. Claude parses the JSON, fixes each failing check, and re-runs verification. Loop until everything passes or iteration limit is hit.

The magic: Verification output is machine-readable JSON, not prose logs. Claude doesn't have to interpret error messages — the JSON tells it exactly what's wrong and where.

Example Verification Output
{ "step": 1, "passed": 8, "failed": 2, "checks": [ { "check": "api_POST_project_exists", "pass": true, "detail": "POST /projects implemented in routes/projects.ts" }, { "check": "typescript_clean", "pass": false, "detail": "Type error on line 45: expected string, got undefined" } ] }
Phase 2

Phase 2: Testing

Verification scripts that catch bugs before they escape
The Verification Pyramid
Build verification scripts that test at multiple levels: file existence (does the code exist?), pattern matching (does it contain the required functions?), compilation (does TypeScript accept it?), tests (do unit tests pass?), behavioral (does it actually work?). The first three catch issues fast. The last one catches subtle bugs.
1
File existence checks
Is the file created? Does it exist at the right path?
2
Pattern matching checks
Does the code contain required functions, exports, or implementations?
3
Compilation checks
npx tsc --noEmit — does TypeScript accept it?
4
Test checks
npm test — do unit tests pass?
5
Behavioral checks
API response time, data persistence, integration with other components.
Phase 2

Phase 2: Integration

When one step depends on another
TaskFlow Phase 2 has multiple steps: API foundation → endpoints → frontend components → database integration → real-time syncing. Each step's verification includes checks that depend on prior steps. The verification script for step 4 (database integration) won't pass if step 3 (API endpoints) didn't complete. Claude respects these dependencies automatically.

Cross-step verification: In CLAUDE.md, mark which steps are prerequisites for other steps. Claude checks dependencies before starting a step.

Phase 2

Phase 2: Error Recovery

What happens when verification fails repeatedly
A check can fail up to 5 times. If it still fails on the 6th attempt, Claude marks it as BLOCKED in PROGRESS.md with full error context, then moves on. This prevents infinite loops on unsolvable problems. The next session's Claude can take a fresh look, or you can escalate to human review.
Bad: Infinite loop
Claude keeps trying to fix a check that's fundamentally broken, burning tokens until the context window fills up.
Good: Graceful failure
After 5 attempts, Claude logs "BLOCKED: API endpoint returns 500 on line 87" and continues to the next step.
Phase 2-3

Phase 2 → Phase 3: Code Complete

From verified to polished
At the end of Phase 2, all verification checks pass. The code works, tests pass, no TypeScript errors. But it's not shipping yet — it's in Phase 3 territory: edge cases, performance, documentation, security hardening. These often take longer than Phase 2 implementation itself.
Phase 3

Phase 3: Polish

From "works" to "great"
The code works. But does it feel good to use? TaskFlow Phase 3: add visual feedback on drag-drop, improve error messages, add loading states, refine animations, polish the UI until it feels snappy. This is where the app becomes a product.

In Phase 3, perfection is cheap. You already have working code. Claude can iterate on UX details, add transitions, improve colors, optimize layouts — all within the same framework.

Phase 3

Phase 3: Performance

Making it fast
Where to Optimize
Phase 3 includes performance review: code splitting, bundle size, API response times, database query optimization, caching strategies. TaskFlow: the board with 1000 tasks should still render smoothly. WebSocket events should sync in under 50ms. These are Phase 3 concerns.
1
Frontend performance
Code splitting, lazy loading, memoization, reducing re-renders.
2
Backend performance
Database indexing, query optimization, caching, rate limiting.
3
Network performance
Compression, batch APIs, reducing payload size.
Phase 3

Phase 3: Documentation

Making it maintainable
Code is read 10× more than it's written. Phase 3 documentation includes: README with setup instructions, API documentation (OpenAPI or similar), architecture decision records (ADRs) for major design choices, contributor guidelines, and inline code comments explaining why (not what).

Claude is great at documentation. It can generate README sections, API docs, architecture diagrams, and decision records from the implemented code. Use Phase 3 to formalize the knowledge you've built.

Phase 3

Phase 3: Edge Cases

Making it robust
What happens when: the network is slow, the user has 10,000 tasks, two users edit the same task simultaneously, the server restarts, the database is temporarily unavailable? Phase 3 is about handling these edge cases gracefully. TaskFlow: conflict resolution for simultaneous edits, graceful degradation when the server is down, proper error messages to the user.
Bad: Crashes on edge cases
App works with 10 tasks, breaks with 1000. Two concurrent edits result in data loss. Error messages are cryptic.
Good: Robust handling
Works with 100K tasks. Concurrent edits are resolved via timestamp-based conflict resolution. Errors are clear and actionable.
Phase 3

Phase 3: Code Quality

Making it sustainable
Code linting, type safety, test coverage targets, accessibility compliance (WCAG), security scanning. Phase 3 is where you enforce quality standards so future maintainers (including future-you) have an easy time. TaskFlow: 90%+ test coverage, zero TypeScript errors even with strict mode, WCAG AA accessibility, no security vulnerabilities in dependencies.
Phase 3

Phase 3: Hardening

Making it production-ready
Pre-Launch Checklist
Before shipping: Security audit (no hardcoded secrets, proper authentication, rate limiting), load testing (can it handle expected traffic?), backup & recovery (what if the database fails?), monitoring & logging (can you debug issues in production?), deployment strategy (blue-green, canary, or rolling?).

Phase 3 is not about speed anymore. It's about confidence. You're removing every possible reason it could fail in production.

Closing

Conclusion: From Spark to Shipped

The complete journey with Claude as your partner
You've seen the three-phase flow: design with Claude in Phase 1, let Claude execute autonomously in Phase 2, refine and harden in Phase 3. The key insight is that each phase has a different goal, and the output of each phase feeds into the next. CLAUDE.md from Phase 1 enables autonomous execution in Phase 2. All checks passing in Phase 2 enables focus on polish in Phase 3.

This isn't magic — it's structure. The patterns work because they're built on how AI actually works: Claude is excellent at synthesis and creativity (Phase 1), excellent at code generation and verification (Phase 2), and excellent at refinement and edge cases (Phase 3). Each phase leverages Claude's strengths.

Next Steps
Start with a small project. Go through Phase 1 deliberately: lock down the design, write CLAUDE.md, define your verification protocol. Then run /build-all and let Claude execute. When all checks pass, move to Phase 3 refinement. The patterns scale — use them on a one-person script or a team project.
