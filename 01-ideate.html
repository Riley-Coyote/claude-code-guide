Section 01

From Spark to Shipped

Building with Claude is different. You're not just writing code — you're orchestrating an AI that generates, tests, fixes, and ships autonomously.
Every great project follows the same arc: you start with a spark of an idea, you explore what's possible, you lock down a design, you build like hell, you test ruthlessly, and then you polish until it shines. The difference with Claude Code is that Claude handles the execution — you focus on the thinking and the decisions.

This is Phase 01 of 03. This guide walks you through the Ideate phase — the conversational phase where you and Claude nail down what you're building. Phase 02 (Cowork & Plan) covers the setup. Phase 03 (Autonomous Build) covers the autonomous execution loop.

Phase 01 (Ideate) is where you think. You and Claude work together to nail down what you're building, the architecture, the constraints, and the exact acceptance criteria. You're in a conversation, not a code editor.

Phase 02 (Cowork & Plan) is where you set up the rails. CLAUDE.md, project anatomy, task briefs, verification scripts, slash commands. You're at the keyboard with Claude Code, building the blueprint for autonomous execution.

Section 03

Everyone Builds Differently

Here's a framework that works
There's no single "right way" to code. But for autonomous Claude execution at scale, a few patterns matter: machine-readable verification (so Claude knows exactly what failed), self-contained task briefs (so Claude doesn't have to hunt for context), cross-session state files (so work persists across context resets), and hard constraints (so Claude doesn't drift).

This phase is where those patterns are born. Every decision you lock down in Ideate becomes a constraint Claude follows autonomously later. Time spent here is time multiplied in Phase 03.

You'll see how to structure it, how Claude builds it autonomously, and how the three phases play out in practice. The patterns scale from a single script to a full-featured application.

Phase 01

Claude as Thinking Partner

The Ideate workflow: you and Claude collaborate to nail down the entire project before any code is written.
Ideate is a conversation. You bring the idea, Claude brings the execution expertise. Together you answer the hard questions: What exactly are we building? What's in scope and what's deferred? What's the architecture? What are the non-negotiables? This phase ends when you have enough clarity to write a CLAUDE.md file — but writing that file itself happens in Phase 02.

Where to have this conversation: Claude chat (claude.ai) is ideal — it's fast, it has good memory, and you're not burning context on code tools you don't need yet. Save Claude Code for when you're ready to build.

Ideate · Step 01

The Spark

From idea to hypothesis
What Happens
Start with a one-sentence problem statement. TaskFlow: "Teams need a fast way to visualize their project workflow." That's it. Claude asks clarifying questions: Who's the user? What's the use case? What problem does it solve? Why can't they use existing tools?
Your Deliverable
A Clear Problem Statement
One paragraph that tells Claude who the users are, what problem they have, and why they can't use existing tools. This becomes the north star for every technical decision that follows.
Ideate · Step 02

The Foundations

From hypothesis to architecture
What Happens
With the problem locked down, Claude asks about the technical foundation. What platform? What language? What database? What integrations? TaskFlow: web app → React frontend + Node.js backend. Postgres for persistence. Real-time via WebSockets. These decisions ripple through the entire build.
Your Deliverable
Tech Stack & Architecture Decisions
The platform, language, key libraries, database choice, and deployment target. Not "maybe use Redis" — concrete decisions that Claude can build toward.
Ideate · Step 03

The Design Lock

From architecture to specification
What Happens
Now Claude asks for the design constraints. What screens? What data model? What API endpoints? What's in Phase 1 vs. Phase 2? TaskFlow v1: login, project creation, drag-drop board, WebSocket sync. No AI recommendations — that's v2. This locked-down scope is critical.
Your Deliverable
Scope & Design Spec
List of screens, API endpoints, data model sketch, and explicit scope boundaries. "What we're NOT building in v1" is as important as what we are.

Exit criteria for Ideate: You can describe the project to a stranger in 60 seconds. You know the stack. You know the scope. You know what "done" means. If any of those are fuzzy, stay in Ideate a little longer — it's cheap here and expensive later.
