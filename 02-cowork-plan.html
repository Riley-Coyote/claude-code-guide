Launch Planning

From mockup to launch criteria
What Happens
What does "done" mean? TaskFlow v1: all endpoints implemented, 85% test coverage, zero TypeScript errors, login works with test users, board renders and syncs in under 200ms. Claude asks for these constraints up front so it knows exactly what to verify against.
Your Deliverable

Explicit criteria: performance targets, test coverage %, API response times, browser compatibility, accessibility standards. Make it measurable.

Project Anatomy

The file structure that makes autonomous execution possible. Every file has a specific role in the feedback loop.
your-project/ ├── CLAUDE.md ← bootstrap: Claude reads this first, always ├── PROGRESS.md ← state: cross-session memory file ├── .claude/ │ ├── settings.json ← pre-approved shell commands │ └── commands/ │ ├── build-all.md ← /build-all slash command │ ├── build-step.md ← /build-step N slash command │ ├── verify.md ← /verify N slash command │ └── resume.md ← /resume slash command ├── docs/tasks/ │ ├── task-01-*.md ← self-contained brief for step 1 │ ├── task-02-*.md ← self-contained brief for step 2 │ └── ... ← one brief per build step ├── scripts/verify/ │ ├── verify-step-01.sh ← machine-readable verification for step 1 │ ├── verify-step-02.sh ← machine-readable verification for step 2 │ ├── ... │ └── verify-all.sh ← master runner: all steps sequentially ├── src/ ← your actual source code └── tests/ ← test suite
Why Each Part Matters
CLAUDE.md is the bootstrap. Claude Code reads it automatically when entering the project directory. It defines what the project is, the code style rules, and the autonomous execution protocol. Without it, every session starts from scratch.

PROGRESS.md is the memory. It tracks which steps are complete, which are blocked, and what happened in each session. Claude reads it at the start of every session and writes to it at the end. This is how state persists across context window resets.

Task briefs are the delegation units. Each brief is self-contained — a subagent can read one brief and have everything it needs to build that step without reading any other file in the project.

Verification scripts are the feedback mechanism. They close the loop by telling Claude exactly what's working and what isn't, in a format it can parse and act on programmatically.
Cowork · Step 04

CLAUDE.md — The Bootstrap File

The single most important file in any autonomous Claude Code project. Read automatically on session start — defines the entire execution contract.
What to Include
1
Project identity
One paragraph on what the project is. Not a marketing pitch — a technical description that tells Claude what it's building and for what platform/runtime.
2
Hard constraints
The non-negotiable rules. "No frameworks in the UI." "TypeScript strict mode." "Dark mode only." "No semicolons." These are the rails that prevent Claude from drifting into default patterns that don't match your project.
3
Code style
Be specific: const vs let, semicolons or not, error handling conventions, comment philosophy ("explain why, not what"). Claude will match whatever style you document — but only if you document it.
4
Build order
A numbered sequence of build steps, each with a brief description and a pointer to its task brief in docs/tasks/. This is the execution plan Claude follows.
5
The autonomous protocol
Document the BUILD → VERIFY → FIX → VERIFY loop explicitly. Tell Claude where verification scripts are, what format they output, how many fix iterations to attempt, and how to update PROGRESS.md. Don't assume it knows — spell it out.
6
What NOT to build
An explicit out-of-scope list prevents Claude from gold-plating. Without it, it may add features that seem logical but aren't wanted.
Example Structure
View — CLAUDE.md skeleton
# Project Name ## What this is One paragraph. Technical description. Platform, runtime, purpose. ## Critical design decisions (do not deviate) - Hard constraint 1 - Hard constraint 2 - Hard constraint 3 ## Code style - TypeScript strict, no semicolons, const over let - Error handling: always catch, always log, never crash - Comments: explain WHY, not WHAT ## Build order Step 1: Scaffold → docs/tasks/task-01-scaffold.md Step 2: Core logic → docs/tasks/task-02-core.md ... ## Autonomous build protocol 1. Read task brief from docs/tasks/task-0N-*.md 2. Build the implementation 3. Run bash scripts/verify/verify-step-0N.sh 4. Parse JSON output — fix each failing check 5. Re-verify (max 5 iterations per check) 6. Update PROGRESS.md 7. Commit ## What NOT to build - Feature X (deferred to v2) - Feature Y (out of scope)
Cowork · Step 05

Task Briefs & Subagent Delegation

The orchestrator/worker pattern. The main Claude session manages state. Subagents do the actual coding. Task briefs are the interface.
The Orchestrator Pattern
Orchestrator
→
reads PROGRESS.md
→
delegates to subagent
→
runs verification
→
updates state
The orchestrator never writes application code itself. It reads state, decides what to build next, spins up a subagent with the task brief, and then verifies the result. This separation matters because subagents get a fresh context window — they can focus entirely on the implementation without carrying the overhead of project-level state management.
What Makes a Good Task Brief
1
Self-contained
The subagent should be able to read ONLY this brief and have everything it needs. Inline the relevant type definitions, API contracts, and file paths. Don't say "see the schema doc" — paste the relevant interfaces directly into the brief.
2
Exact file paths
Tell the subagent exactly which files to create or modify. Don't leave directory structure decisions to the subagent — that's an orchestrator decision.
3
Verification criteria inline
Include what "done" means within the brief itself. "The adapter must: intercept fetch calls, parse SSE streams, deduplicate via SHA-256, and pass npx tsc --noEmit." The subagent should know if it succeeded before the orchestrator even runs verification.
4
Constraints repeated
Always append the code style and hard constraints to every brief. Subagents don't read CLAUDE.md. They only see what you give them. If you don't say "no semicolons," they'll add semicolons.
Delegation Template
View — Subagent prompt pattern
Delegate to subagent with this brief: "Build [component name] for [project name]. ## What to build [Exact description of the component] ## Files to create/modify - Create: src/path/to/new-file.ts - Modify: src/path/to/existing-file.ts (add X to Y) ## Interfaces (inlined from schema) [Paste the relevant TypeScript interfaces here] ## Acceptance criteria - [ ] Criterion 1 - [ ] Criterion 2 - [ ] npx tsc --noEmit exits clean - [ ] npm test passes ## Constraints Code style: no semicolons, TypeScript strict, const over let, never var. [Any other hard constraints from CLAUDE.md]"
Cowork · Step 06

Custom Slash Commands

Markdown files in .claude/commands/ become slash commands. They're the user interface for your autonomous build system.
The Four Essential Commands
command
/build-all
The "press start and walk away" command. Reads PROGRESS.md, finds the first non-PASSED step, and executes the full build loop for every remaining step. Delegates each step to a subagent, runs verification, fixes failures, updates progress, commits. Stops only when all steps pass or a step is BLOCKED.
command
/build-step N
Builds a single step. Same loop as /build-all but scoped to one step. Useful for iterating on a specific component without touching anything else. Checks that prerequisite steps are PASSED before starting.
command
/verify N
Runs verification for step N (or "all" for every step). Doesn't build anything — just checks the current state. Use this to confirm a step is still passing after unrelated changes, or to get a quick status check.
command
/resume
The session-start command. Reads PROGRESS.md, identifies where work left off, runs a regression check on the last passed step, then continues. This is how you bridge sessions — the new Claude picks up exactly where the last one stopped.
Pre-Approved Permissions
The .claude/settings.json file pre-approves shell commands so Claude doesn't pause for permission on every npm test or bash scripts/verify/.... Without this, autonomous execution breaks every time Claude needs to run a command.
View — settings.json
{ "permissions": { "allow": [ "Bash(npm *)", "Bash(npx *)", "Bash(node *)", "Bash(git *)", "Bash(bash scripts/*)", "Bash(cat *)", "Bash(ls *)", "Bash(find *)", "Bash(mkdir *)", "Bash(cp *)", "Bash(rm *)", "Bash(grep *)", "Bash(head *)", "Bash(tail *)", "Bash(wc *)", "Bash(tsc *)" ] } }

The Handoff

From design to code generation
At the end of Phase 02, you give Claude one instruction: /build-all. That slash command exists in .claude/commands/build-all.md. It tells Claude to read CLAUDE.md, read PROGRESS.md, and start executing the build order step by step. Phase 03 is fully autonomous.

Key insight: Phase 01 is about decisions. Phase 02 is about setup. Phase 03 is about execution. The handoff is clean — you've decided and prepared everything, Claude executes everything. No back-and-forth needed unless something actually breaks.
