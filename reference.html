Part 04 · Slash Commands

Every custom command the guide defines.

Four build commands you wire up in Phase 02, and four quality gates you trigger in Phase 03.
command what it does defined in
/build-all Press-start: reads PROGRESS.md, walks through every remaining build step autonomously. Guide 02 · Slash Commands
/build-step N Build a single step (N). Same loop scoped to one task brief. Guide 02 · Slash Commands
/verify N Runs verification for step N only. Use all for every step. Doesn't build. Guide 02 · Slash Commands
/resume Session-start: reads PROGRESS.md, runs a regression check on the last passed step, then continues. Guide 02 · Slash Commands
/review Code review gate. Runs on the diff before committing. Catches bugs and security issues. Guide 03 · Quality Gates
/accessibility WCAG 2.1 AA audit. Run on any step that produces HTML. Catches contrast, ARIA, keyboard nav. Guide 03 · Quality Gates
/architecture Generates an Architecture Decision Record for significant design choices. Run after integration steps. Guide 03 · Quality Gates
/testing-strategy Validates test coverage follows the testing pyramid (mostly unit, some integration, few E2E). Guide 03 · Quality Gates
Part 05 · Key Files

The files Claude needs to run autonomously.

If it's here, Claude reads it. If it's not here, Claude doesn't know about it.
path role covered in
CLAUDE.md Bootstrap file. Project identity, hard constraints, code style, build order, autonomous protocol. Read automatically at session start. Guide 02 · CLAUDE.md Bootstrap
PROGRESS.md Cross-session memory. Step status table, current state, blockers log, dated session log. Read at start, written at end. Guide 03 · Cross-Session State
.claude/settings.json Pre-approved shell commands. Prevents permission prompts during autonomous execution. Guide 02 · Slash Commands
.claude/commands/*.md Custom slash command definitions. One markdown file per command. Guide 02 · Slash Commands
docs/tasks/task-0N-*.md Self-contained task briefs. Each brief has exact scope, file paths, inlined schemas, acceptance criteria. One per build step. Guide 02 · Task Briefs
scripts/verify/verify-step-0N.sh Per-step verification scripts. JSON output: passed/failed checks with specific detail strings. Guide 03 · Machine-Readable Verification
scripts/verify/verify-all.sh Master verification runner. Runs every step sequentially, stops at first failure. Guide 03 · Machine-Readable Verification
Part 06 · Core Vocabulary

The terminology the guide uses.

Every named concept, in one place, with a one-line definition.
term definition where covered
Core Loop The five-stage cycle every autonomous session runs: BUILD → VERIFY → FIX → VERIFY → COMMIT. Guide 03 · s1
Orchestrator The main Claude session. Reads state, delegates to subagents, runs verification. Never writes application code itself. Guide 02 · s5
Subagent A child Claude process with its own fresh context window. Reads one task brief, writes code, returns. Guide 02 · s5
Task Brief A self-contained spec that a subagent reads once and implements completely. Inlines types, paths, constraints, acceptance criteria. Guide 02 · s5
Machine-Readable Verification Verification scripts that output structured JSON with pass/fail per check. Lets Claude parse failures and fix them programmatically. Guide 03 · s4
Quality Gate A non-structural review wired into specific steps. Catches bugs, security, accessibility, architecture issues that verification misses. Guide 03 · s9
Iteration Cap Maximum 5 fix attempts per failing check. If still failing, mark BLOCKED and move on. Prevents infinite loops. Guide 03 · s1
BLOCKED Status for a check that failed five times. Logged in PROGRESS.md with full error context. A fresh session or human can revisit. Guide 03 · s7
Cross-Session State The mechanism — PROGRESS.md — by which one session's work persists to the next across context-window resets. Guide 03 · s8
Context Window Economics The mental model for spending tokens. Self-contained briefs, specific verification, subagent delegation, pre-approved commands — all reduce waste. Guide 03 · s11
The Handoff The Phase 02 → Phase 03 transition. You type /build-all. Claude takes over. No more human-driven editing until something breaks. Guide 02 · s8
Polish Phase Post-verified work: edge cases, performance, accessibility, documentation. Runs the same loop with different task briefs. Guide 03 · s13
Hardening Pre-launch audit: security, load testing, backup, monitoring, deployment strategy. Removes every reason it could fail in production. Guide 03 · s18
Design Lock The Phase 01 step where scope, data model, screens, and API are fixed. Everything downstream builds against this. Guide 01 · s8

Part 09 · Anti-Patterns

What not to do, each one discovered the hard way.

Five shortcuts that feel reasonable in the moment and degrade autonomous execution quality every time.
Prose verification
"Check it works." Claude interprets vaguely, different sessions interpret differently, no reproducibility. Always JSON with specific detail strings.
Monolithic sessions
Building the whole app in one session. Context fills up, quality degrades, if the session crashes you lose everything. Step + verify + commit every time.
Orchestrator writes code
Main session writes implementation AND manages state. Context fills with details, leaving less room for orchestration. Delegate to subagents.
Memory dump on session start
"Tell me everything you know about this project." Wastes 20-40% of context before any work begins. Just run /resume and state your intent.
Infinite fix loops
No iteration cap. Claude tries the same fix 20 times, burning the whole context window on one stubborn check. Five attempts, then BLOCKED.
Full treatment
Every anti-pattern is covered in depth in Guide 03 · Anti-Patterns with side-by-side comparisons and the right alternative.

Part 11 · Golden Rules

The five rules that govern everything.

When in doubt, apply one of these.
Golden Rules

1. CLAUDE.md is the bootstrap. PROGRESS.md is the memory. Verification scripts are the feedback. Task briefs are the delegation interface. Everything else is implementation detail.

2. The orchestrator manages state. Subagents write code. Never blur these roles.

3. Machine-readable verification is the single most important investment. Every hour spent writing specific checks saves ten hours of Claude debugging vague failures.

4. State your intent on session start. Let CLAUDE.md and PROGRESS.md provide the context. Don't front-load with memory dumps.

5. Five fix attempts, then BLOCKED. Never let one issue consume the whole session.
