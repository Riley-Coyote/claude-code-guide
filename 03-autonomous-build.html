Section 01

The Core Loop

Every autonomous Claude Code session runs on one fundamental cycle. Everything else in this guide exists to make this loop faster, more reliable, and self-correcting.
BUILD
→
VERIFY
→
FIX
→
VERIFY
→
COMMIT
BUILD — Claude reads a task brief and writes the implementation. For complex tasks, it delegates to subagents.

VERIFY — A verification script runs and returns structured JSON: every check is pass: true or pass: false with a human-readable detail string explaining what's wrong.

FIX — Claude parses the JSON, identifies failing checks, and fixes each one. Maximum 5 attempts per individual check.

VERIFY (again) — Re-run verification until all checks pass or the iteration limit is hit.

COMMIT — Once all checks pass, stage changes and commit with a descriptive message. Update the progress file.

Why this works: The key insight is that the verification output is machine-readable. Claude doesn't need to interpret log files or guess what went wrong — the JSON tells it exactly which check failed and why. This turns debugging from a creative task into a mechanical one.

The Iteration Cap
Five attempts per failing check is the sweet spot. Below that, you give up too early on solvable issues. Above that, you're burning tokens on something that probably needs human input. If a check fails 5 times, mark it as BLOCKED in the progress file with full error context, then move on. Don't let one stuck check halt the entire pipeline.
Build · Step 01

Code Generation

The BUILD phase of the loop: Claude receives a task brief and generates implementation
What Happens
Claude reads the first task brief from docs/tasks/task-01-*.md. The brief is self-contained: here's what to build, here are the file paths, here are the constraints, here's what "done" looks like. Claude generates the full implementation, then moves to verification.
Why Task Briefs Matter
A task brief is not a casual request. It's a formal specification that a subagent Claude instance reads once and implements completely. Include: exact scope, file paths, relevant type definitions inlined, constraints from CLAUDE.md, and acceptance criteria. The subagent shouldn't need to read anything else.

The VERIFY → FIX → VERIFY cycle in action

After implementing, Claude runs a verification script: bash scripts/verify/verify-step-01.sh. The script outputs structured JSON: which checks passed, which failed, and why. Claude parses the JSON, fixes each failing check, and re-runs verification. Loop until everything passes or iteration limit is hit.

The magic: Verification output is machine-readable JSON, not prose logs. Claude doesn't have to interpret error messages — the JSON tells it exactly what's wrong and where.

Example Verification Output
{ "step": 1, "passed": 8, "failed": 2, "checks": [ { "check": "api_POST_project_exists", "pass": true, "detail": "POST /projects implemented in routes/projects.ts" }, { "check": "typescript_clean", "pass": false, "detail": "Type error on line 45: expected string, got undefined" } ] }
Section 04

Machine-Readable Verification

The innovation that makes the recursive loop work. Verification scripts that output structured JSON instead of prose — so Claude can parse failures and fix them programmatically.
Output Format
{ "step": 3, "name": "claude-adapter", "total": 12, "passed": 10, "failed": 2, "checks": [ { "check": "sse_parser_exists", "pass": true, "detail": "SSE parser found in claude-inject.ts" }, { "check": "dedup_cache", "pass": false, "detail": "No SHA-256 hashing found for deduplication" }, { "check": "typescript_clean", "pass": false, "detail": "2 type errors in claude-inject.ts lines 45, 112" } ] }
Writing Effective Checks
Each check tests one specific thing. The detail field is the key — it must tell Claude exactly what's wrong and where to look. Vague details like "test failed" force Claude to investigate. Specific details like "No SHA-256 hashing found for deduplication in claude-inject.ts" let Claude go straight to the fix.
View — Bash verification pattern
#!/bin/bash # verify-step-03.sh — Claude adapter verification CHECKS="[]" PASS_COUNT=0 FAIL_COUNT=0 check() { local name="$1" pass="$2" detail="$3" if [ "$pass" = "true" ]; then PASS_COUNT=$((PASS_COUNT + 1)) else FAIL_COUNT=$((FAIL_COUNT + 1)) fi CHECKS=$(echo "$CHECKS" | jq ". + [{\"check\":\"$name\",\"pass\":$pass,\"detail\":\"$detail\"}]") } # File existence [ -f "src/adapters/claude-inject.ts" ] \ && check "inject_exists" "true" "Page inject script found" \ || check "inject_exists" "false" "Missing src/adapters/claude-inject.ts" # Pattern matching grep -q "SHA-256\|crypto.subtle.digest" src/adapters/claude-inject.ts \ && check "dedup_hash" "true" "Content hashing found" \ || check "dedup_hash" "false" "No SHA-256 hashing in claude-inject.ts" # TypeScript compilation TSC_OUT=$(npx tsc --noEmit 2>&1) if [ $? -eq 0 ]; then check "typescript_clean" "true" "Zero type errors" else ERR=$(echo "$TSC_OUT" | head -3 | tr '\n' ' ') check "typescript_clean" "false" "$ERR" fi # Output final JSON echo "{\"step\":3,\"name\":\"claude-adapter\",\"total\":$((PASS_COUNT+FAIL_COUNT)),\"passed\":$PASS_COUNT,\"failed\":$FAIL_COUNT,\"checks\":$CHECKS}"
Build · Step 03

The Verification Pyramid

Verification scripts that catch bugs before they escape
Build verification scripts that test at multiple levels: file existence (does the code exist?), pattern matching (does it contain the required functions?), compilation (does TypeScript accept it?), tests (do unit tests pass?), behavioral (does it actually work?). The first three catch issues fast. The last one catches subtle bugs.
1
File existence checks
Is the file created? Does it exist at the right path?
2
Pattern matching checks
Does the code contain required functions, exports, or implementations?
3
Compilation checks
npx tsc --noEmit — does TypeScript accept it?
4
Test checks
npm test — do unit tests pass?
5
Behavioral checks
API response time, data persistence, integration with other components.
Build · Step 04

Integration & Dependencies

When one step depends on another
A real project has multiple steps: API foundation → endpoints → frontend components → database integration → real-time syncing. Each step's verification includes checks that depend on prior steps. The verification script for step 4 (database integration) won't pass if step 3 (API endpoints) didn't complete. Claude respects these dependencies automatically.

Cross-step verification: In CLAUDE.md, mark which steps are prerequisites for other steps. Claude checks dependencies before starting a step.

Build · Step 05

Error Recovery

What happens when verification fails repeatedly
A check can fail up to 5 times. If it still fails on the 6th attempt, Claude marks it as BLOCKED in PROGRESS.md with full error context, then moves on. This prevents infinite loops on unsolvable problems. The next session's Claude can take a fresh look, or you can escalate to human review.
Bad: Infinite loop
Claude keeps trying to fix a check that's fundamentally broken, burning tokens until the context window fills up.
Good: Graceful failure
After 5 attempts, Claude logs "BLOCKED: API endpoint returns 500 on line 87" and continues to the next step.
Section 08

Cross-Session State

Claude Code sessions have finite context windows. PROGRESS.md is the file that makes state persist across sessions so each new Claude picks up where the last left off.
PROGRESS.md Structure
The file has four sections, each serving a specific purpose:
Step Status Table
A markdown table with one row per build step: step number, name, status (NOT_STARTED / IN_PROGRESS / PASSED / BLOCKED), check counts, timestamp, and notes. Claude reads this first to know where things stand.
Current State Section
A snapshot of build health right now: TypeScript errors, test counts, build status, known environment issues. Updated every session.
Blockers Section
Any check that failed 5 times gets logged here with full error context. The next session's Claude can either solve it with fresh perspective or escalate to the human.
Session Log
Chronological entries: what was done, what was fixed, what decisions were made. Each entry has a date and a brief summary. This is the audit trail.

Critical rule: Always read PROGRESS.md at session start. Always write to it at session end. If you skip the write, the next session loses everything this session accomplished.

Section 09

Quality Gates

Verification scripts catch structural issues. Quality gates catch everything else — security flaws, performance problems, accessibility failures, and architectural missteps.
Gate Schedule
Wire quality gates into specific steps where they matter most. Don't run every gate on every step — that wastes tokens. Match the gate to the type of work:
/review — After every step
Run a code review on the diff before committing. Catches bugs, security flaws, and maintainability issues that verification scripts miss because they only check structure, not logic.
/accessibility — After UI steps
WCAG 2.1 AA compliance check. Run on any step that produces HTML. Catches contrast ratio failures, missing ARIA attributes, keyboard navigation gaps.
/architecture — After integration steps
Generate an Architecture Decision Record (ADR) for significant design choices. Especially valuable when the step involves choosing between multiple approaches (which database, which protocol, which API pattern).
/testing-strategy — After test steps
Validate that test coverage follows the testing pyramid: mostly unit tests, some integration tests, few end-to-end tests. Catches the common mistake of writing only happy-path tests.
Section 10

Session Management

How to start, run, and end Claude Code sessions for maximum effectiveness with minimum wasted context.
Starting a Session
Wasteful
"Retrieve all your memory about this project and summarize what we've done so far before starting any work."

Burns 20-40% of context on a memory dump that may not be relevant to today's task.
Effective
"Run /resume" or "Run the production readiness prompt in docs/tasks/production-readiness-prompt.md"

CLAUDE.md loads automatically. PROGRESS.md has the state. Claude pulls only the context it needs for the specific task.
The rule: If CLAUDE.md and PROGRESS.md can orient the session, just state your intent. Only front-load context retrieval when the relevant decisions live in your memory systems (past conversations, external docs) and not in the repo.
During a Session
Let Claude work. The autonomous loop is designed to run without intervention. Check in on PROGRESS.md periodically if you want to see status, but don't interrupt the loop unless something is actually wrong. Each interruption costs context — Claude has to re-orient after every user message.
Ending a Session
If the session runs out of context or you need to stop it, the last thing to verify is that PROGRESS.md has been updated. If it hasn't, ask Claude to update it before closing. The next session depends entirely on that file being accurate.
Section 11

Context Window Economics

Every token in the context window is a finite resource. The patterns in this guide are designed to maximize useful work per token spent.
Where Tokens Go
In an autonomous build session, tokens distribute roughly like this: ~15% on reading state and task briefs, ~50% on code generation, ~20% on verification and fix loops, ~15% on state management and commits. The fix loop is where waste happens — if verification is vague, Claude spends tokens investigating instead of fixing.
Cost-Saving Patterns
1
Self-contained task briefs eliminate re-reading
If the subagent has to read 5 files to understand its task, that's 5× the context cost. A single brief that inlines everything means one read.
2
Specific verification details eliminate investigation
"No SHA-256 hashing found in claude-inject.ts" → Claude adds SHA-256 hashing. "Test failed" → Claude has to read logs, find the failing test, understand the error, then fix. The second path costs 5-10× more tokens.
3
Subagent delegation amortizes context
The orchestrator maintains project-level state in a long-lived context. Subagents get fresh windows focused on single tasks. This prevents the orchestrator's context from filling up with implementation details it doesn't need to retain.
4
Pre-approved commands eliminate round-trips
Every permission prompt costs a message round-trip. Pre-approving common commands in settings.json lets Claude run npm, tsc, git, and verification scripts without stopping.

Polish · Intro

Code Complete → Polish

From verified to polished
At this point, all verification checks pass. The code works, tests pass, no TypeScript errors. But it's not shipping yet. What's left is the polish loop: edge cases, performance, documentation, security hardening. These often take longer than the initial implementation — and the autonomous loop handles them too.
Polish · Step 01

Polish

From "works" to "great"
The code works. But does it feel good to use? Polish means: visual feedback on interactions, improved error messages, loading states, refined animations, UI that feels snappy. This is where the app becomes a product.

In Polish, perfection is cheap. You already have working code. Claude can iterate on UX details, add transitions, improve colors, optimize layouts — all within the same verification framework.

Polish · Step 02

Performance

Making it fast
Where to Optimize
Performance review covers: code splitting, bundle size, API response times, database query optimization, caching strategies. The board with 1,000 tasks should still render smoothly. WebSocket events should sync in under 50ms. These are Polish concerns — once correctness is locked in.
1
Frontend performance
Code splitting, lazy loading, memoization, reducing re-renders.
2
Backend performance
Database indexing, query optimization, caching, rate limiting.
3
Network performance
Compression, batch APIs, reducing payload size.
Polish · Step 03

Documentation

Making it maintainable
Code is read 10× more than it's written. Documentation includes: README with setup instructions, API documentation (OpenAPI or similar), architecture decision records (ADRs) for major design choices, contributor guidelines, and inline code comments explaining why (not what).

Claude is great at documentation. It can generate README sections, API docs, architecture diagrams, and decision records from the implemented code. Use Polish to formalize the knowledge you've built.

Polish · Step 04

Edge Cases

Making it robust
What happens when: the network is slow, the user has 10,000 tasks, two users edit the same task simultaneously, the server restarts, the database is temporarily unavailable? Polish is about handling these edge cases gracefully — conflict resolution for simultaneous edits, graceful degradation when the server is down, proper error messages to the user.
Bad: Crashes on edge cases
App works with 10 tasks, breaks with 1000. Two concurrent edits result in data loss. Error messages are cryptic.
Good: Robust handling
Works with 100K tasks. Concurrent edits are resolved via timestamp-based conflict resolution. Errors are clear and actionable.
Polish · Step 05

Code Quality

Making it sustainable
Code linting, type safety, test coverage targets, accessibility compliance (WCAG), security scanning. This is where you enforce quality standards so future maintainers (including future-you) have an easy time. Targets: 90%+ test coverage, zero TypeScript errors even with strict mode, WCAG AA accessibility, no security vulnerabilities in dependencies.
Polish · Step 06

Hardening

Making it production-ready
Pre-Launch Checklist
Before shipping: Security audit (no hardcoded secrets, proper authentication, rate limiting), load testing (can it handle expected traffic?), backup & recovery (what if the database fails?), monitoring & logging (can you debug issues in production?), deployment strategy (blue-green, canary, or rolling?).

Polish is not about speed anymore. It's about confidence. You're removing every possible reason it could fail in production.

Section 19

Anti-Patterns

Common mistakes that degrade autonomous execution quality. Each one was discovered the hard way.
Prose verification
"Check that the adapter works correctly and handles edge cases." Claude has to interpret what "works correctly" means. Different sessions interpret it differently. No reproducibility.
JSON verification
20 specific pass/fail checks with detail strings. Same result every run. Claude can parse and act on failures without interpretation.
Monolithic sessions
"Build the entire app in one session." Context fills up. Quality degrades. If the session crashes, everything is lost unless PROGRESS.md was being updated.
Step-by-step with commits
Each step is verified and committed independently. If a session dies mid-step, the completed steps are preserved. The next session resumes cleanly.
Orchestrator writes code
The main session writes implementation code AND manages state. Its context fills with code details, leaving less room for orchestration decisions and fix iterations.
Orchestrator delegates
Main session only manages state and verification. Subagents write code in fresh contexts. The orchestrator stays lean and can run more fix iterations.
Memory dump on session start
"Tell me everything you know about this project." Wastes 20-40% of context on potentially irrelevant history before any work begins.
Intent-first session start
"Run /resume" or state the specific task. Claude reads CLAUDE.md and PROGRESS.md automatically — it pulls only the context needed for the actual work.
Infinite fix loops
No iteration cap on failing checks. Claude tries the same fix 20 times, burning the entire context window on one stubborn issue.
5-iteration cap with BLOCKED
After 5 attempts, mark as BLOCKED with full context in PROGRESS.md. Move on. A fresh session (or human) can solve it with new perspective.

Golden Rules

1. CLAUDE.md is the bootstrap. PROGRESS.md is the memory. Verification scripts are the feedback. Task briefs are the delegation interface. Everything else is implementation detail.

2. The orchestrator manages state. Subagents write code. Never blur these roles.

3. Machine-readable verification is the single most important investment. Every hour spent writing specific checks saves ten hours of Claude debugging vague failures.

4. State your intent on session start. Let CLAUDE.md and PROGRESS.md provide the context. Don't front-load with memory dumps.

5. Five fix attempts, then BLOCKED. Never let one issue consume the whole session.
